# Codex Collaboration Rules

## Purpose
이 프로젝트는 "리뷰가 형식적/보상유도형인지, 진심 리뷰 (`genuine`)인지 분류"하는 AI/NLP 포트폴리오 프로젝트다.

핵심 원칙은 "빠르게 완성"이 아니라 "사용자가 모든 코드를 직접 리뷰하고 이해할 수 있게 만드는 것"이다.

## Mandatory Working Style
모든 Codex 세션은 아래 원칙을 반드시 따른다.

1. 코드를 한 번에 크게 만들지 않는다.
2. 파일 1개 또는 함수 1~3개 단위로 잘라서 작업한다.
3. 각 단계마다 왜 이 코드가 필요한지 설명한다.
4. 사용자가 코드 하나하나를 리뷰할 수 있도록 작은 단위로 진행한다.
5. 사용자가 확인하기 전 다음 큰 단계로 넘어가지 않는다.
6. "그냥 돌아가게 만들기"보다 "사용자가 설명 가능한 구조"를 우선한다.
7. 복잡한 추상화, 과도한 최적화, 불필요한 매직 코드는 피한다.

## Explanation Standard
코드를 작성하거나 수정할 때마다 최소한 아래 내용을 설명한다.

- 이 파일/함수의 역할
- 입력과 출력
- 왜 이런 구조로 짰는지
- 다른 선택지와 비교했을 때 이유
- 사용자가 특히 봐야 할 리뷰 포인트

가능하면 면접에서 어떻게 설명할지도 덧붙인다.

## Project Scope
현재 프로젝트 목표는 다음과 같다.

- 한국어 리뷰 데이터를 사용한다.
- 목표는 "좋아요성 리뷰 (`good`)" vs "진심 리뷰 (`genuine`)" 분류다.
- 단순 플랫폼 구분기가 되지 않도록 주의한다.
- 라벨은 플랫폼이 아니라 리뷰 내용 기준으로 정한다.
- 리뷰이벤트 여부, 재주문 횟수, 별점 같은 메타데이터는 정답 라벨 기준이 아니라 보조 feature 후보로 다룬다.

## Labeling Principles
라벨링은 아래 기준을 기본으로 한다.

### Label 0: 진심 리뷰 (`genuine`)
- 메뉴, 상품, 배송, 맛, 품질, 사용 경험 등 구체 정보가 있다.
- 실제 경험 서술이 중심이다.
- 이벤트 문구가 있더라도 내용이 충분히 구체적이면 정보성으로 볼 수 있다.

### Label 1: 좋아요성 리뷰 (`good`)
- 이벤트 참여 문구가 중심이다.
- 칭찬 위주지만 구체 경험 정보가 거의 없다.
- 매우 짧거나 템플릿성 표현이 많다.
- 길어도 템플릿처럼 반복되고 정보 밀도가 낮으면 형식적 리뷰로 본다.

### Exclude / Uncertain
- 애매한 샘플은 억지로 2분류에 넣지 않는다.
- 판단이 어려운 데이터는 `exclude` 또는 `uncertain`으로 따로 관리한다.

## Dataset Rules
- 플랫폼별 문체 차이가 크므로 플랫폼 편향을 항상 점검한다.
- 플랫폼명, 노골적인 이벤트 문구, 메타데이터만으로 정답을 맞추는 구조를 경계한다.
- 가능하면 텍스트-only baseline과 텍스트+메타데이터 모델을 분리 비교한다.
- train/test split 시 데이터 누수와 플랫폼 편향을 점검한다.
- 가능하면 플랫폼 홀드아웃 평가도 검토한다.

## Implementation Order
구현은 아래 순서를 기본으로 한다.

1. 라벨링 기준 문서화
2. CSV/데이터 스키마 확정
3. 데이터 로딩 코드
4. 전처리 코드
5. baseline 모델(TF-IDF + Logistic Regression 등)
6. 평가 코드(F1, precision, recall, confusion matrix)
7. 오분류 분석
8. 고급 모델(BERT 계열 파인튜닝 등)

## Coding Style
- 코드 길이는 짧고 읽기 쉽게 유지한다.
- 함수는 역할이 분명해야 한다.
- 변수명은 직관적으로 짓는다.
- 파일을 나누는 이유가 명확해야 한다.
- 주석은 최소한으로, 꼭 필요한 맥락만 적는다.
- 사용자가 Python/ML 학습용으로 읽는다는 점을 항상 고려한다.

## What To Avoid
- 한 번에 대규모 코드 생성
- 설명 없는 리팩터링
- 사용자가 검토하기 어려운 긴 파일
- 근거 없는 모델 선택
- accuracy만 제시하는 평가
- 데이터 누수
- 플랫폼별 규칙을 그대로 정답 라벨로 사용하는 설계

## Review-First Workflow
모든 세션은 가능하면 아래 흐름을 따른다.

1. 이번 단계 목표를 짧게 설명
2. 작은 단위 코드 작성
3. 코드 설명
4. 사용자가 리뷰할 포인트 제시
5. 다음 단계 진행 여부 확인

## Success Criteria
이 프로젝트의 성공 기준은 다음과 같다.

- 사용자가 각 파일과 함수의 역할을 설명할 수 있다.
- baseline부터 고급 모델까지 성능 비교가 가능하다.
- 라벨링 기준과 데이터 한계를 명확히 설명할 수 있다.
- 포트폴리오와 면접에서 "내가 이해하고 만든 프로젝트"로 설명 가능하다.
