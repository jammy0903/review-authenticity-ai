# Project Structure

## Purpose
이 문서는 `review-authenticity-ai` 프로젝트의 폴더 구조와 각 파일의 존재 이유를 설명한다.

목표는 구조를 단순하게 유지하면서도,
사용자가 각 파일이 왜 있는지 직접 이해하고 리뷰할 수 있게 만드는 것이다.

## Recommended Structure

```text
review-authenticity-ai/
├─ rules/
│  ├─ CODEX_COLLAB_RULES.md
│  ├─ LABELING_GUIDE.md
│  ├─ CSV_SCHEMA.md
│  └─ PROJECT_STRUCTURE.md
├─ data/
│  ├─ raw/
│  ├─ labeled/
│  └─ processed/
├─ notebooks/
├─ src/
│  ├─ __init__.py
│  ├─ config.py
│  ├─ data_loader.py
│  ├─ preprocess.py
│  ├─ features.py
│  ├─ train_baseline.py
│  ├─ evaluate.py
│  └─ error_analysis.py
├─ outputs/
│  ├─ models/
│  ├─ metrics/
│  └─ figures/
├─ tests/
├─ README.md
├─ requirements.txt
└─ .gitignore
```

## Folder-by-Folder Explanation

### `rules/`
프로젝트 운영 기준 문서 폴더다.

왜 필요한가:
- 어떤 Codex 세션이 오더라도 바로 프로젝트 원칙을 이해할 수 있다.
- 라벨링 기준이 흔들리지 않게 유지할 수 있다.
- 코드보다 먼저 봐야 할 합의사항을 문서로 고정할 수 있다.

### `data/`
데이터 파일을 저장하는 최상위 폴더다.

왜 필요한가:
- 원본 데이터와 가공 데이터를 섞지 않기 위해서다.
- 데이터 흐름을 추적하기 쉽게 만들기 위해서다.
- 실험 재현성을 높이기 위해서다.

### `data/raw/`
원본 리뷰 데이터를 저장한다.

왜 필요한가:
- 처음 수집한 데이터를 보존하기 위해서다.
- 전처리 실수나 라벨링 오류가 생겼을 때 원본으로 돌아갈 수 있다.
- "원본"과 "가공본"을 분리해야 데이터 관리가 쉬워진다.

### `data/labeled/`
사람이 직접 라벨링한 CSV를 저장한다.

왜 필요한가:
- 이 프로젝트의 핵심 자산은 라벨링된 데이터이기 때문이다.
- 어떤 기준으로 정답을 만들었는지 추적 가능해야 한다.
- 학습 전 단계에서 사람이 검토하기 가장 쉬운 데이터가 이 폴더에 모인다.

### `data/processed/`
전처리 후 모델 학습에 바로 쓸 수 있는 데이터를 저장한다.

왜 필요한가:
- 모델 입력용 데이터를 원본과 분리하기 위해서다.
- 전처리 규칙을 바꿔도 결과 파일을 다시 만들 수 있게 하기 위해서다.
- 실험마다 어떤 데이터 버전을 썼는지 구분하기 위해서다.

### `notebooks/`
EDA, 샘플 탐색, 빠른 실험용 노트북을 둔다.

왜 필요한가:
- 데이터 분포를 확인하거나 빠르게 실험할 때 편하다.
- 다만 핵심 로직은 `src/`에 두고, 노트북은 보조 용도로만 써야 한다.
- 노트북만 남기면 나중에 코드 구조를 이해하기 어려워진다.

### `src/`
실제 파이썬 코드가 들어가는 폴더다.

왜 필요한가:
- 핵심 로직을 노트북과 분리하기 위해서다.
- 파일별 책임을 나눠서 리뷰와 테스트를 쉽게 하기 위해서다.
- baseline에서 고급 모델로 확장할 때 구조가 망가지지 않게 하기 위해서다.

### `outputs/`
모델 결과물을 저장한다.

왜 필요한가:
- 학습된 모델, 평가 지표, 시각화 결과를 코드와 분리하기 위해서다.
- 어떤 실험에서 어떤 결과가 나왔는지 정리하기 쉽다.
- 포트폴리오 정리 시 결과물만 따로 꺼내 보기 좋다.

### `outputs/models/`
학습된 모델 파일을 저장한다.

왜 필요한가:
- baseline 모델과 이후 모델을 버전별로 저장할 수 있다.
- 추론이나 재평가를 할 때 다시 학습하지 않아도 된다.

### `outputs/metrics/`
평가 결과 파일을 저장한다.

왜 필요한가:
- F1, precision, recall, confusion matrix 같은 결과를 기록할 수 있다.
- 모델별 성능 비교표를 만들기 쉽다.

### `outputs/figures/`
시각화 결과를 저장한다.

왜 필요한가:
- confusion matrix 그림, 텍스트 길이 분포, 클래스 분포 차트를 관리할 수 있다.
- README나 포트폴리오 문서에 활용하기 쉽다.

### `tests/`
테스트 코드를 둔다.

왜 필요한가:
- 데이터 로딩, 컬럼 검증, 전처리 로직처럼 작은 코드도 깨질 수 있기 때문이다.
- 나중에 리팩터링해도 안전하게 바꿀 수 있다.

## File-by-File Explanation

### `README.md`
프로젝트 소개 문서다.

왜 필요한가:
- 프로젝트 목적, 데이터, 실행 방법, 결과를 한눈에 보여준다.
- 포트폴리오에서 가장 먼저 읽게 되는 문서다.

### `requirements.txt`
필요한 파이썬 패키지 목록이다.

왜 필요한가:
- 누구든 같은 환경을 다시 만들 수 있게 한다.
- 설치 과정을 단순하게 만든다.

### `.gitignore`
추적하지 않을 파일 목록을 정의한다.

왜 필요한가:
- 가상환경, 캐시, 모델 파일, 대용량 결과물이 Git에 섞이는 걸 막는다.
- 저장소를 깔끔하게 유지한다.

### `src/__init__.py`
`src`를 파이썬 패키지로 인식하게 한다.

왜 필요한가:
- 다른 파일에서 모듈 import가 쉬워진다.
- 구조가 커질 때 확장성이 좋아진다.

### `src/config.py`
경로와 기본 설정을 모은다.

왜 필요한가:
- 데이터 경로, 출력 경로, 랜덤 시드 같은 공통 설정을 한곳에 모을 수 있다.
- 여러 파일에서 같은 경로를 중복 하드코딩하지 않게 해준다.

### `src/data_loader.py`
CSV를 읽고 기본 검증을 수행한다.

왜 필요한가:
- 필수 컬럼 존재 여부를 초기에 확인할 수 있다.
- 데이터 로딩 책임을 한 파일에 모아 구조를 단순하게 할 수 있다.
- 이후 학습 코드가 데이터 읽기 세부사항에 덜 의존하게 된다.

### `src/preprocess.py`
텍스트 전처리 로직을 둔다.

왜 필요한가:
- 공백 정리, 결측치 처리, 기본 정규화 같은 작업을 분리할 수 있다.
- 전처리 규칙을 바꾸더라도 학습 코드와 분리되어 관리가 쉽다.

### `src/features.py`
baseline용 feature 생성 로직을 둔다.

왜 필요한가:
- 텍스트 길이, 이벤트 문구 포함 여부, 특정 키워드 포함 여부 등을 실험할 수 있다.
- 텍스트-only 모델과 텍스트+메타데이터 모델을 비교할 때 유용하다.

### `src/train_baseline.py`
baseline 모델을 학습한다.

왜 필요한가:
- 가장 먼저 비교 기준을 만들어야 이후 개선이 의미가 생긴다.
- TF-IDF + Logistic Regression 같은 단순하고 해석 가능한 모델을 구현하기 좋다.

### `src/evaluate.py`
평가 지표 계산과 결과 저장을 담당한다.

왜 필요한가:
- 학습 코드와 평가 코드를 분리하면 구조가 더 명확해진다.
- F1, precision, recall, confusion matrix를 일관되게 계산할 수 있다.

### `src/error_analysis.py`
오분류 샘플을 분석한다.

왜 필요한가:
- 모델이 어디서 틀리는지 봐야 다음 개선 방향이 나온다.
- 포트폴리오에서 "숫자만 본 게 아니라 실제 오류를 분석했다"는 근거가 된다.

## Recommended Build Order
처음부터 모든 파일을 만들 필요는 없다.

권장 순서는 아래와 같다.

1. `README.md`
2. `data/labeled/` 아래 라벨링 템플릿 CSV
3. `src/config.py`
4. `src/data_loader.py`
5. `src/preprocess.py`
6. `src/train_baseline.py`
7. `src/evaluate.py`
8. `src/error_analysis.py`
9. 필요 시 `src/features.py`

## Review Principle
이 구조의 핵심 목적은 파일 수를 늘리는 것이 아니라,
사용자가 각 파일의 책임과 흐름을 직접 리뷰할 수 있게 만드는 것이다.

파일이 늘어났는데 책임이 불분명하면 구조가 나빠진 것이다.
파일 수가 적더라도 역할이 명확하면 그 구조가 더 좋다.
